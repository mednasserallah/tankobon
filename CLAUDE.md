# CLAUDE.md ‚Äî Tankobon

## What this project is

**Tankobon** is an unofficial fork of [Mihon](https://github.com/mihonapp/mihon) (which is itself the successor to Tachiyomi), an Android manga reader written in Kotlin and licensed under **Apache-2.0**. This fork narrows Mihon's scope to a **local-file-only, volume-based** manga reader: it will read manga you already own on-device, organized by *volume* rather than *chapter*, with no remote sources or extensions.

## Roadmap (NOT YET DONE ‚Äî future work)

The fork's goals:

1. **Remove the extension system entirely.** ‚úÖ **DONE** (Task 2, on `feature/remove-extensions`). No remote sources, no extension repo/installer/updater, no online browsing/search, no download subsystem. `LocalSource` (id `0L`) is now the only source the app can hold. See the "Extension System Map" section below for exactly what was removed vs. deferred.
2. **Local source rework ‚Äî volumes, not chapters.** ‚úÖ **DONE** (Task 3, on `feature/volume-based-migration`). The reading unit is now a **volume**: `Chapter`‚Üí`Volume` renamed throughout data/domain/UI, `chapter_number REAL`‚Üí`volume_number INTEGER`, `chapters` table‚Üí`volumes`, new `volumes.volume_number_end` + `mangas.edition` columns (migration v14), and a new `VolumeRecognition` parser for the naming convention below. Verified end-to-end on device. See "Chapter ‚Üí Volume Migration Map" below. The parser (`source-local/.../LocalSource.kt` `getVolumeList`) treats each `.cbz`/`.zip`/`.cbr`/folder as one volume and uses this opinionated naming convention:
   ```
   Series Name/
     ‚îú‚îÄ Series Name - Volume 01 (Year).cbz
     ‚îú‚îÄ Series Name - Volume 02 (Year).cbz
     ‚îú‚îÄ cover.jpg        (optional, ignore ‚Äî not a volume)
     ‚îî‚îÄ details.json     (optional, ignore ‚Äî not a volume)
   ```
   Edge cases to handle: single-volume/one-shots with no "Volume NN" (`Boy Meets Maria (2021).cbz`); series names containing parentheses (`BLAME! (Master Edition)`) must not be confused with the trailing `(Year)`; `cover.jpg`/`details.json` sidecars must never be parsed as volumes; empty series folders are skipped, not errored.
   Start point: `source-local/src/androidMain/kotlin/tachiyomi/source/local/LocalSource.kt`.

This was the last major planned roadmap item. Remaining work is polish/backlog only (see "Deferred cleanup").

Note: pre-existing entries scanned by the old build keep their old-parser numbers until re-scanned (library update skips entries with unread volumes by default) ‚Äî fresh scans are correct.

### Decisions recorded (documented convention)
- **One-shot / single-volume** (no `Volume NN` segment): treated as **volume 1**; the volume's display name is the file's own title (year stripped), e.g. `Boy Meets Maria (2021).cbz` ‚Üí volume 1 named "Boy Meets Maria". (`SyncVolumesWithSource` guards against `VolumeSanitizer` blanking the name when it equals the series title.)
- **Volume-count badge** counts **volumes** (the library unread badge = number of volume rows). An omnibus `Volume 01-02` is one reading unit (one row) but represents volumes 1‚Äì2; the badge counts the row. *(The volumes-represented count is available via `volume_number_end` for a future "owned volumes" display if desired.)*
- **`edition`** lives on the manga/series (one edition per folder), null when absent; shown as a subtitle badge on the manga detail header.

Phase 0 investigation results (kept for reference) below. The goal: model the reading unit as a **volume** (one `.cbz`/`.zip`/`.cbr`/folder = one volume) instead of a chapter, and rewrite the local file parser for Tankobon's naming convention.

### Phase 0 ‚Äî where "chapter" lives (investigated, verified)

Scope: **197 files** reference "chapter" (case-insensitive) ‚Äî `app` 131, `domain` 35, `data` 22, `source-api` 5, `source-local` 2, others. A full type-rename is **high-churn and high-risk** (SQLDelight-generated code + live external trackers + backup proto). See the Phase 2 decision below.

1. **Data layer.** `Chapter` entity = `domain/.../tachiyomi/domain/chapter/model/Chapter.kt` (`chapterNumber: Double`, `read`, `bookmark`, `lastPageRead`, `sourceOrder`, `url`, `name`, ‚Ä¶). Table = `data/.../tachiyomi/data/chapters.sq` (`chapter_number REAL NOT NULL`, FK `manga_id ‚Üí mangas`). Mapped positionally by SQLDelight ‚Üí `ChapterMapper`/`copyFromSChapter` (`eu/kanade/domain/chapter/model/ChapterMapper.kt`). `chapter_number` is `REAL/Double` to allow decimals like "10.5" ‚Äî **volumes are whole numbers, so this is a semantic simplification, not just a rename.**
2. **Local parser.** `source-local/.../LocalSource.kt` `getChapterList()` builds one `SChapter` per file, name = filename-without-extension, and `chapter_number = ChapterRecognition.parseChapterNumber(...)` (`domain/.../chapter/service/ChapterRecognition.kt` ‚Äî generic Tachiyomi "Ch. 12"/"Vol.1 Ch.4.5" regex). Files are sorted by **name, natural order, descending** (`compareToCaseInsensitiveNaturalOrder`). Sidecars are currently NOT excluded beyond dot-files (json/cover would be filtered only by format check). **This is what Phase 1 replaces.** Note: `SyncChaptersWithSource` (`app/.../domain/chapter/interactor/`) *re-runs* `parseChapterNumber` but preserves an already-recognized (`‚â•0`) number ‚Äî so a number set by the parser survives.
3. **Progress tracking.** Per-`chapter` row: `read`/`bookmark`/`lastPageRead`. `ChapterGetNextUnread` + `getChapterSort` (`domain/.../chapter/service/ChapterSort.kt`: SOURCE / NUMBER / UPLOAD_DATE / ALPHABET). No chapter-*count*-specific logic that breaks under volumes; "next unread" is just first-unread in sorted order. Carries over conceptually to per-volume.
4. **Reader UI.** `ReaderViewModel` builds prev/next **positionally** from `chapterList.sortedWith(getChapterSort(...))` ‚Äî so correct numeric sort (Phase 1) makes "next volume" work with no structural change. Transition card = `presentation/reader/ChapterTransition.kt` (shows `chapter.name` + a "missing N chapters" gap via `MissingChapters.calculateChapterGap`, which does `floor(a)-floor(b)-1` ‚Äî **already integer-based**, harmless for whole volumes; it now means "missing volumes"). Decimal-specific display: `presentation/util/ChapterNumberFormatter.kt` (`DecimalFormat("#.###")`).
5. **Library / manga-detail UI.** Volume-list rows: `presentation/manga/MangaScreen.kt:696` ‚Üí `display_mode_chapter` string with `formatChapterNumber(chapterNumber)` when `displayMode == CHAPTER_DISPLAY_NUMBER`, else raw `name`. Missing-count separators in `MangaScreenModel.chapterListItems`. Library unread badge = count of unread chapter rows (becomes volume count). All terminology + the `#.###` formatter are Phase 4.
6. **Trackers / sync ‚Äî ‚öÝÔ∏è SURPRISE: STILL LIVE, NOT dead.** All **6 generic trackers** (MAL, Anilist, Kitsu, Shikimori, Bangumi, MangaUpdates) survived the extension removal and **actively push "last chapter read" (a number) to external services** (`data/track/*Api.kt`, `TrackChapter`, `AddTracks`, `RefreshTracks`, `DelayedTrackingUpdateJob`). `manga_sync.last_chapter_read` is `REAL/Double`; `Track.lastChapterRead: Double`. Only `SyncChapterProgressWithTrack` (remote‚Üílocal) is a dead no-op. **Implication:** retyping/renaming chapter-number would ripple into 6 external API serializers + the `manga_sync` schema + backup `BackupTracking` proto. This is the single biggest argument for Option B. Kept as-is; the number now semantically carries a *volume* count to services that expect a chapter count ‚Äî an imperfect-but-acceptable mapping, flagged for a future deliberate decision (some services have distinct volume/chapter fields).
7. **Backup / restore ‚Äî proto format constrains us.** `BackupChapter.chapterNumber` = `@ProtoNumber(9)` **`Float`**; restore matches chapters **by `url`** (not by number), manga by url+source. So: renaming the DB column does NOT break restoring old Mihon/Tankobon backups (matching is url-based), **but** `BackupChapter.chapterNumber(9)` must stay `Float` at proto field 9 ‚Äî do not retype it. New fields go under fresh ProtoNumbers (backup format is unversioned but uses default `ProtoBuf`, which tolerates unknown fields): `BackupChapter` next free = **14**; `BackupManga` next free = **113**.

### Phase 2 decision ‚Äî **Option A (full rename)** ‚Äî chosen by the user

Phase 0 recommended Option B (semantic reuse) because of the deep coupling below, but the **user explicitly chose Option A (full rename)** after that risk was surfaced. So: rename **`Chapter` ‚Üí `Volume`** and **`chapter_number` ‚Üí `volume_number` (integer)** throughout the data layer ‚Äî domain entity, `chapters` table ‚Üí `volumes` table, SQLDelight queries/views/generated code, mappers, `SChapter` ‚Üí `SVolume`, reader/library identifiers, and the tracker/`manga_sync` layer.

**Non-negotiable safety constraints (kept even under "full rename"):**
- **Backup proto field *numbers* are wire format ‚Äî never renumber.** Kotlin property names may change (`chapterNumber` ‚Üí `volumeNumber`) but the `@ProtoNumber(9)` stays `9` and stays `Float`, so old Mihon/Tankobon backups still restore (matching is url-based anyway). New fields use fresh numbers: `BackupVolume` next free = **14**, `BackupManga` edition = **113**.
- **The entire tracker / `manga_sync` layer keeps its "chapter" terminology ‚Äî it models EXTERNAL services, not the local reading unit.** MAL/Anilist/Kitsu/Shikimori/Bangumi/MangaUpdates fundamentally track "chapters read" / "total chapters"; their JSON keys (`num_chapters_read`, `progress`, `chapters`, `ep_status`, `chapter`) AND the internal columns/fields that mirror them (`manga_sync.last_chapter_read`, `total_chapters`, `Track.lastChapterRead`, `TrackChapter`, etc.) stay as-is. Renaming them to "volume" would be an active lie about what those services store. The value pushed is now the local *volume* number mapped onto a chapter-based remote field ‚Äî an imperfect but accepted mapping (some services have distinct volume fields; deferred as a future product decision). This bounds the rename to the local reading-unit model and avoids a risky `manga_sync` migration + touching 6 external API serializers.
- **Migrations are additive / non-destructive** ‚Äî table/column renames use `ALTER TABLE ‚Ä¶ RENAME` (SQLite preserves data), and new columns default `NULL`. No user data destroyed on upgrade.
- **One bounded exception (documented):** Kotlin **package *directory paths*** are kept (`tachiyomi.domain.chapter.*`, `eu.kanade.domain.chapter.*`) ‚Äî only the **class names** inside them rename (`Chapter`‚Üí`Volume`, so `import ‚Ä¶chapter.model.Chapter` ‚Üí `import ‚Ä¶chapter.model.Volume`). Renaming physical directories/packages is pure churn (every import path changes) with zero user- or schema-visible benefit, whereas renaming the *types* delivers the readability goal. Physical DB identifiers for the **local reading unit** ARE renamed (`chapters`‚Üí`volumes`, `chapter_number`‚Üí`volume_number`, `history.chapter_id`‚Üí`volume_id`). The `manga_sync` (external-tracker) table is left untouched per the constraint above.

**Concrete data-model changes (real migrations ‚Äî central to the task):**
- `chapter_number REAL` ‚Üí `volume_number INTEGER` (whole-number; the Phase 1 parser only emits integers). Retyped at the DB, domain (`Volume.volumeNumber: Long`), mapper, backup (`BackupVolume.volumeNumber` stays `@ProtoNumber(9) Float` for compat, converted at the boundary), and the reader/UI.
- **NEW `volumes.volume_number_end INTEGER` (nullable)** ‚Äî set only for omnibus range files (`Volume 01-02` ‚Üí `volume_number = 1`, `volume_number_end = 2`); `null` for single-volume files. Sorting + next/prev key off `volume_number` (the start). Plumbed: schema + migration, `Volume` model, SQLDelight mappers, `SVolume` + `copyFromSVolume`, `BackupVolume @ProtoNumber(14)`.
- **NEW `mangas.edition TEXT` (nullable)** ‚Äî the extracted edition tag (e.g. `"Omnibus Edition"`), a **series/folder-level** property (one edition per folder), `null` when absent. Plumbed: `mangas.sq` + migration, `Manga` model, `MangaMapper` (all overloads), `SManga` + `toDomainManga`, `BackupManga @ProtoNumber(113)`.
- **DB migrations from v13**: additive columns (`volume_number_end`, `edition`) + the `chapters‚Üívolumes` / `chapter_number‚Üívolume_number` renames, expressed as non-destructive `.sqm` steps.

### Naming-convention parser rules (Phase 1 spec ‚Äî `VolumeRecognition`, in `domain/.../volume/service/` after the rename)

Input is a **series folder name** and a **volume file name** (basename, extension stripped). Rules:
- **Year** = the **last** parenthetical group matching `^\d{4}$`. This disambiguates trailing `(2016)` from an edition like `(Master Edition)`. Parsed **per file** (never inherited across volumes).
- **Edition** (folder-level) = any parenthetical group in the *series folder name* that is **not** a 4-digit year, captured verbatim (`Gantz (Omnibus Edition)` ‚Üí title `Gantz`, edition `Omnibus Edition`). No hardcoded edition whitelist.
- **Volume number** = `Volume\s+(\d+)(?:-(\d+))?` (case-insensitive), capturing an optional range end. `Volume 01-02` ‚Üí start `1`, end `2`, display `Volume 01-02` (range shown verbatim, not collapsed). Sort by **start**.
- **Single-volume / one-shot** (no "Volume NN" segment, e.g. `Boy Meets Maria (2021)`): treated as **Volume 1** (convention: assign volume number `1`, `volume_number_end = null`; display uses the file's own title text). *(Documented choice: number = 1 so it sorts and tracks like any single volume.)*
- **Sidecars excluded**: `cover.*`, `details.json`, `ComicInfo.xml`, `.noxml`, dot-files, and any non-archive/non-directory/non-epub file are never volumes.
- **Empty series folder** (no valid volume files): produces an empty volume list ‚Äî skipped gracefully, no error/crash.
- **Sort**: numeric by volume start (`1, 2, ‚Ä¶ 9, 10`), never lexical.
- **Volume-count badge**: counts **volumes represented** (an omnibus `Volume 01-02` counts as 2), not files ‚Äî better reflects what the user owns.
- **Fallback** (filename matches nothing above): use the whole basename as the display title, volume number `1`, and log it (don't crash / don't silently drop).

### Detection rule (documented, section 3 of the task spec)
The cover is **the alphanumerically-first image in the archive/folder, using natural sort** ‚Äî no `[Cover]`-tag / `p000` / `cover.*`-name special-casing (scan-group conventions vary too much; natural-sort-first already resolves both real examples correctly). Zero images ‚áí no thumbnail (graceful, not a crash). Implemented as the pure, unit-tested `VolumeCoverSelector.selectCover(names)` in `core/common`.

### Reading-order algorithm (Phase 2 ‚Äî `ReadingOrderSorter`, pure/unit-tested)
Given a flat list of detected lines (text + `Rect` bounding box) and the manga's RTL/LTR flag:
1. **Cluster into rows** by vertical overlap ‚Äî two lines share a row when their y-ranges overlap by more than a tolerance fraction of the shorter box height (tolerance tuned so a staggered layout doesn't split every line into its own row, nor merge genuinely separate bands).
2. **Order within a row** by x: right-to-left for `RIGHT_TO_LEFT`, else left-to-right.
3. **Order rows** top-to-bottom (by row's min-y).
Unit tests cover: single-column page, two side-by-side panels needing RTL ordering, and an irregular/staggered layout. Detection uses ML Kit **`Line`** granularity (verified against real pages to be the most natural per-bubble grouping; `Block` can merge separate bubbles).

**English-only / Arabic-only for now**, architected to extend (source & target languages are parameters, not hardcoded constants in the OCR/translate call sites).

### Refinement pass (post-Task 6, same branch) ‚Äî detection pipeline
The OCR pipeline in `openTextDetectionDialog` is now: `recognize ‚Üí TextLineMerger.merge (incl. hyphen collapse) ‚Üí SentenceCase.normalize ‚Üí drop number-only lines (isNumberOnlyLine) ‚Üí ReadingOrderSorter.sort`. `isNumberOnlyLine` filters lines that contain a digit and no letters (page numbers bleeding into the scan, number-only bubbles) ‚Äî they carry no translatable text; a line with any letter ("Chapter 1") is kept.

1. **Bubble/paragraph merge** (`TextLineMerger`, pure + unit-tested). ML Kit `Line`s split wrapped text in one bubble ("WHO IS" / "THAT?!"). Two lines merge only when the vertical gap between them ‚â§ `0.6 √ó average line height` (wrapped-line leading, **not** bubble-to-bubble spacing) **and** their horizontal ranges overlap ‚â• `0.35 √ó` the narrower width (same text column, not two side-by-side bubbles). Thresholds are deliberately conservative ‚Äî under-merging (a split bubble) is preferred over blending two unrelated bubbles into nonsense. Merged text is joined with a single space, then hyphenated line breaks are resolved (see round 2, item 1).
2. **Sentence-casing** (`SentenceCase`, pure + unit-tested). Applied after merge, before display AND translation. Lowercases, capitalizes the first letter of the string and the first letter after `.`/`!`/`?`, and restores standalone `"i"`‚Üí`"I"`. It's a readability heuristic, not grammar: **known limitations** ‚Äî acronyms/initialisms are lowercased ("NASA"‚Üí"Nasa"), proper nouns lose their capital unless sentence-initial, an abbreviation's internal period ("Mr.") spuriously capitalizes the next word, and an ellipsis `"..."` is treated as a sentence end (so `"the black... swords-man..."` ‚Üí `"The black... Swords- man.."`).
3. **Per-line translate.** Each line has a Translate icon beside its Copy icon, calling the same `translateLine(index)` used by "Translate all" (reusing the language-pack download flow). The per-line button is disabled while that line is downloading/translating, and "Translate all" only targets `Idle`/`Error` lines ‚Äî so the two paths never double-translate a line.
4. **Zoom-scoped detection.** When zoomed past fit-to-screen on a **paged** viewer, OCR runs only on the visible region: `ReaderPageImageView.getVisibleImageRegion()` converts the SSIV viewport (`viewToSourceCoord`) to source-image pixels and returns null at ~fit zoom (‚â•95% of the image visible) or for non-zoomable/not-ready images; `PagerViewer` exposes it through `Viewer.getCurrentPageVisibleRegion()`; `ReaderViewModel` reads it on the main thread before dispatching OCR; `decodePageBitmap` then decodes just that rect via `BitmapRegionDecoder`. At default zoom behaviour is byte-for-byte unchanged (whole page). **Webtoon** returns null (whole page) ‚Äî continuous-scroll cropping was out of scope. Because the visible region fills more of the 2048px decode budget, small text is effectively higher-resolution, which should also improve OCR accuracy. Verified: default-zoom detection unchanged on device. The zoomed-crop path could not be exercised end-to-end on the emulator (SELinux blocks `sendevent` multi-touch and injected taps don't trigger a pinch/double-tap-zoom gesture), so it is verified by construction + the robust null/clamp guards; not a code-reachability problem (the zoom state is reached cleanly via `state.viewer`).

### Refinement pass 2 (post-Task 6, same branch) ‚Äî hyphens, editing, confidence
1. **Hyphenated line-break resolution** (`TextLineMerger.collapseHyphenation`, pure + unit-tested; runs inside merge, before sentence-casing). When a merged line has a `fragment-` followed by whitespace + `nextWord`, two manga patterns are distinguished heuristically (no dictionary): a **stutter** (`"PR- PREPARE"`‚Üí`"PREPARE"`) when `fragment.length ‚â§ 3` **and** `nextWord` starts with `fragment` (case-insensitive) ‚Äî drop the fragment; otherwise **word-wrap** (`"ACCOM- PLISH"`‚Üí`"ACCOMPLISH"`, also fixes the old `"swords- man"`‚Üí`"swordsman"`) ‚Äî splice with no space/hyphen. Applied repeatedly so `"P- P- PREPARE"`‚Üí`"PREPARE"`; a dangling hyphen is left as-is. **Known limitation:** a genuinely-short real hyphenated fragment that happens to prefix the next word is misread as a stutter and dropped ‚Äî rare, accepted, not worth a dictionary. Verified live: the real "ACCOM-/PLISH" bubble in Berserk ‚Üí "‚Ä¶accomplish‚Ä¶".
2. **Editable lines** (`DetectedLineItem.text` + `updateLineText`). Each row is tap-to-edit (OutlinedTextField, confirm via check icon or IME Done). Copy and translate (per-line and translate-all) act on `item.text` (the current, possibly-edited value), so a correction flows through to both ‚Äî **verified**: editing "K"‚Üí"KELLO" then translating produced the Arabic of "KELLO", not "K". An edit commits via `updateLineText`, which also clears that line's stale translation; other lines are untouched and edits persist for the sheet session.
3. **Confidence flagging.** ML Kit's per-line confidence is captured on `DetectedTextLine.confidence` (a merged line takes the **worst** member's, via `min`). Lines below **`LOW_CONFIDENCE_THRESHOLD = 0.5f`** (in `DetectedTextLine.kt`, tunable; 0.5 chosen because clean lettering scores ~0.9+ while smudged/small/stylised text drops below it) are flagged with **dimmed text** (`onSurfaceVariant`) ‚Äî a subtle hint to edit, never a block on copy/translate. The flag clears once the line is edited (`item.text != item.line.text`). Verified: a stray low-confidence "K" detection showed the flag.

### Discovered pre-existing bug (worked around, NOT fixed ‚Äî flagged)
`mihon.core.archive.ArchiveInputStream.read(b, off, len)` (`core/archive/`) is buggy for partial reads: it does `ByteBuffer.wrap(b, off, len)` (capacity = `b.size`) then the private `read()` calls `buffer.clear()`, which resets the limit to the **full capacity**, ignoring `off`/`len`. libarchive can then write past `len` and the method can return a length `> len`. The reader's normal decode path never hits this (it reads whole segments, off=0/len=capacity, where `clear()` is harmless), but `BitmapFactory.decodeStream`'s incremental peek/rewind reads do ‚Äî Skia's `GetPrimitiveArrayRegion` copies the over-length result into a smaller native buffer ‚Üí **native SIGSEGV**. `decodePageBitmap` avoids it by reading the whole entry into a `ByteArray` first (safe off=0 path) and decoding via `decodeByteArray`. A proper fix to `ArchiveInputStream` (respect `off`/`len`, e.g. `ByteBuffer.wrap(b, off, len)` without `clear()`, or slice) is a separate core-archive task ‚Äî left unfixed here to keep this feature scoped and avoid a reader-wide change.
